## 通用执行模型

关系发现采用有向探索，而不是完整数据索引：

1. 盘点所有文件、表、行列维度和表头。
2. 从规模小、角色明确或与用户目标接近的文件提取种子。
3. 用种子精确指纹探测大表候选列。
4. 只从命中行扩展下一跳键。
5. 同时为候选列保留固定数量的 bottom-k 指纹，补充发现未被种子覆盖的高重叠关系。

输入规模可以增长，但持久化状态受以下参数约束：`max_seed_values`、`seed_values_per_column`、`profile_size`、每关系最多 12 条展开证据。

## 种子和候选列

种子文件根据通用信号选择：

- 表格估算单元格数不超过 `seed_cell_budget`。
- 文档或未知行数表格文件大小不超过 `seed_file_bytes`。
- 文件名包含规则、政策、配置、字典、映射、模板、样例、结果及对应英文角色词时提高优先级。
- 没有小文件时，从最小表的前 `bootstrap_rows` 行生成引导种子，但不把该表视为完整扫描。

候选列由通用表头语义识别：ID、编号、编码、代码、父级/来源/目标引用、邮箱、电话、URL、名称和标题。行业名词只作为这些通用后缀的 base，例如“订单ID”和“就诊ID”都使用相同算法。

## 关系判定

| 类型 | 最低证据 | 判定 |
|---|---|---|
| `exact_duplicate` | 完整文件 SHA-256 一致 | `confirmed` |
| `explicit_reference` | 文档明确出现目标文件名 | `confirmed` |
| `seeded_value_link` | 有界种子与候选列规范化值的 SHA-256 精确一致，且列语义兼容或值高特异 | 同表头/同 base 且值中高特异，或同类列且值高特异时为 `confirmed`；低语义兼容时为 `supported_hypothesis` |
| `profiled_value_overlap` | 两个兼容列的 bottom-k 摘要出现精确共享指纹 | 同表头/同 base 至少两个强值，或同类高特异列至少三个强值时为 `confirmed`；仅同属“代码/编号”不自动确认 |

列名相似本身只用于确定扫描范围，不能生成关系。短数字、空值、日期、布尔值和普通名称被过滤或降低特异度。

## 覆盖语义

- `complete`：所有可解析表的候选列均完成扫描；输出覆盖既定关系类型，但不声称覆盖任意自然语言语义。
- `partial`：达到时间预算；`progress.json` 记录表和行偏移，重复执行原命令继续。
- `metadata_only`：只完成文件哈希或表结构，不能据此声称文件无关系。
- 所有输出关系都有精确指纹证据；未进入种子或 bottom-k 摘要的未命中值不声称已覆盖。

“所有关系”应解释为：在已声明候选列和证据规则内发现的全部达标关系，而不是无法验证的任意语义联想。

## 性能边界

- XLSX 首选 Rust Calamine 驱动的 `fastexcel`，只载入候选列；DuckDB `read_xlsx` 作为可用环境下的第二路径，最终使用 `openpyxl` 只读流式模式兜底。
- CSV/TSV、JSONL 和 SQLite 使用流式行读取；Parquet 使用 DuckDB 列式读取。
- 不对大型结构化表拼接整行文本，不运行全行实体正则，不持久化全部 distinct 值。
- 大表即使没有种子命中，也只产生固定大小的列摘要。
- 最终 SQLite 使用 `journal_mode=DELETE`，只保存关系和有限证据，不产生与输入规模线性增长的 WAL。
