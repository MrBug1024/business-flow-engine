# Project objectives

I hope to reverse engineer and replicate the complete business flow based on business historical data, such as having three or more business tables, one business rule table, and a result table ... wait.
Based on these tables, we can accurately deduce the correlation between these tables and how the entire business process is completed from business data to business results. After replicating the entire business process, we hope to use one or more Skills or Tools sets to replicate the business implementation process, so that the Skills or Tools sets have the ability to implement this business process.
When new business table data is passed in, it can produce results with the same structure and logic.

---

Note 2:
In historical data tables, there is a large amount of data, with thousands or even tens of thousands of entries being common. Therefore, never let AI open the data table on its own and look at it one by one.
I suggest that AI call system tools or skills to obtain the header of each table and randomly select 1-3 pieces of data.
---
The implementation is not limited to any algorithm model (such as using decision tree model algorithms, regression model algorithms, etc. if necessary).

---
Note 3:
We should actually distill the capabilities of business scenarios like an AI model, and assign these capabilities to Skill skills or tools, which can then be used to do real business.
---

---
Note 4:
It is often mistaken, for example, the corresponding results in historical business data are obtained based on fields A, B, and C in the business data table. After deduction, it is found that the ability of the business scenario is only a dead rule logic that copies fields A, B, and C to obtain the results,
But in reality, the ability of business scenarios should not be singular. Historical data is only a product of business scenarios and cannot fully represent the full capabilities of the business scenario. For example, a MySQL engineer who creates a database and queries a piece of data cannot be said to only have the ability to create a database and query that data, right? Actually, this engineer
Knowing which data tables are in the database, which fields are in the tables, which fields are keys, how tables are directly related, and so on, this project can query other data, as well as add, delete, modify, and query. The whole thing we need to do is the same principle. Creating a business scenario is never specifically for querying a certain piece of data. It should be like an engineer, knowing which tables, fields, and how to associate them. Based on these abilities, we can do whatever we want
