# Business Capability Reverse Engineering & Generalizable Skill Engine – Specification v2.0

## 1. Project Vision
We are building a system that, given a set of historical business data tables, a business rule table (knowledge base), and one or more historical result tables (examples), reverse engineers the **entire business capability** and packages it into a reusable, parameterized Skill.  
The Skill must behave like a domain expert engineer: it knows the data schema, field semantics, business rules, and can execute any audit/query within that domain **without re-analyzing data** at runtime.

## 2. Core Principles (NEVER violate)
1. **AI as Architect, NOT Data Worker** – The AI model (MiniMax-M3) must NEVER see raw data rows beyond 1–3 samples per table. All heavy data access goes through Python scripts that return only metadata and tiny samples to the AI.
2. **Generalization is Mandatory** – The historical result table is a *validation sample* for one rule only. The final Skill must support ALL rules present in the rule table (and extensible to new ones).
3. **Deterministic, AI-Free Execution** – Once the Skill is built, its runtime execution must NOT call the AI model. It runs SQL templates and Python logic in seconds.
4. **Output Schema Contract** – The Skill's output must always match the structure (columns, types) of the historical result table, unless the user explicitly requests otherwise.

## 4. Detailed Derivation Workflow (Mandatory Phases)

### Phase 0: Business Process Discovery
**Goal:** Document what the business scenario actually *does*.
- AI must ask the user to describe the business context.
- Output: `business_process.md` containing:
  - Plain-language description of the business problem.
  - Steps: what data comes in, how it is processed, what rules are applied, what results are produced.
  - Identification of *input tables*, *rule/standard tables*, *result tables*.
  - A Mermaid flowchart of the process.
- **Gate:** User must explicitly approve this document before continuing.

### Phase 1: Schema & Relationship Extraction
**AI uses tool `extract_metadata(file_paths)`** to get for each table:
- Column names, data types.
- For each column: non-null %, approx. unique count, 1–3 sample values (random).
- Candidate primary keys and foreign keys based on naming and value overlap.
- For the rule table: a structured summary of each rule (violation type, keywords extracted from description, example text trimmed).
**AI then:**
- Proposes table relationships (JOIN keys) and field semantics (what each field means).
- Presents an ER diagram (Mermaid) and a data dictionary draft.
- Asks user to confirm/correct. This must use **structured prompts** (see Section 5).

### Phase 2: Rule Logic Template Extraction
**Goal:** Transform the rule table's natural language entries into executable logic templates for **every rule type**.
- AI reads rule summaries (not full text initially) and requests more detail via `get_rule_detail(index)` if needed.
- For each rule type, AI generates a **rule template** containing:
  - `violation_type` (string)
  - `required_tables` and `join_conditions`
  - `filter_logic` (conditions, thresholds)
  - `aggregation` (GROUP BY, HAVING, etc.)
  - `output_columns` mapping
- These templates are stored in `rule_templates.json`.
- AI must ask clarifying questions if a rule's logic is ambiguous (e.g., “what defines ‘same encounter’?”).
- **Important:** If a rule requires data not present (e.g., a price standard table), the template must mark it as `missing_data: true` and explain what is needed.

### Phase 3: Validate with Historical Result
1. User uploads the historical result table (CSV/Excel).
2. AI selects the rule template that corresponds to that result (e.g., “duplicate charges”).
3. AI generates the **complete SQL query** (or Python logic) for that template.
4. AI calls **`execute_and_compare(sql, result_table)`** – this tool runs the SQL on full historical data, compares with the result file, and returns:
   - `matched_rows`, `missing_rows`, `extra_rows`, and up to 3 sample differences.
5. AI iterates by refining logic until 100% match (or acceptable threshold).
6. Once matched, the validated SQL is stored as the **verified implementation** for that rule.

### Phase 4: Generalization & Complete Rule Library
- AI must now produce SQL templates for **all remaining rules** in `rule_templates.json`, even those without historical results.
- If AI cannot produce a reliable template due to missing data or unclear logic, it must flag it and propose a solution (e.g., “need price table”).
- AI updates `rule_templates.json` with `status: verified|unverified|blocked`.

### Phase 5: Skill Packaging (The “Engineer’s Toolbox”)
The final Skill is a self-contained Python package (or Tool set) with:
- `domain_knowledge.json`: Data dictionary, ER relationships, field semantics, result schema (exact column definitions matching historical result).
- `rule_templates.json`: All rule templates and their SQL implementations.
- `skill_executor.py`: A class that exposes:
  - `list_audit_types()` → returns list of violation types.
  - `execute_audit(violation_type, new_data_folder)` → loads new data (same table names), validates schema, executes the SQL template, and returns a DataFrame **with columns exactly as defined in `domain_knowledge.json`**.
- **Runtime constraint:** `execute_audit` must NOT call any AI model. It must be pure Python/SQL. Execution time should be under 10 seconds for up to 1 million rows (use DuckDB).

## 5. Frontend Interaction Protocol (Mandatory)
- The frontend must render Markdown properly (use a library like `marked.js`).
- When the AI needs user input, it must NOT embed questions in flowing text. Instead, it must output a **structured interaction block**:
  
```json
{
    "interaction": {
      "type": "choice" | "confirm" | "input",
      "question": "Do these table relationships look correct?",
      "options": ["a) Yes, they are correct", "b) No, I will correct them"],
      "allow_custom": true
    }
  }
```

- The frontend renders this as a form. User response is sent back as structured data.

- This mechanism must be used for Phase 0 approval, relationship confirmation, rule ambiguity resolution, etc.

## 6. Algorithmic Assistance (Decision Trees, Regression, Symbolic Regression)
These are optional diagnostic aids to be used only during derivation when AI struggles to infer logic from samples alone. They must be packaged as internal tools:
 `suggest_classification_rules(target_field, feature_fields, table)`

- Fits a decision tree (max_depth=3) on the sample to suggest rule conditions.

- Returns a human-readable rule like `IF field_A > 100 AND field_B == 'X' THEN target = 'Violation'`.

- AI can then translate this into SQL.

`suggest_formula(target_numeric, feature_fields, table)`
- Uses symbolic regression (e.g., gplearn) to find an algebraic formula mapping features to target.

- Returns candidate formulas ranked by error.

`find_anomalies(table, rule_template)`
- Runs an Isolation Forest to detect rows that are outliers, which might indicate missing rules or edge cases.

Important: These tools are never part of the final Skill. They only produce candidate logic for AI to review, translate into deterministic SQL, and validate.

- Optional ML: scikit-learn (decision tree), gplearn (symbolic regression)

## 8. Acceptance Criteria (How to Verify Implementation)
1. Run the derivation on the healthcare audit scenario. It must produce:

   - business_process.md (approved by user)

   - domain_knowledge.json with correct schema and relationships

   - rule_templates.json covering all rule types from the rule table

   - At least one rule validated 100% against historical result.

2. After derivation, delete the historical data. Upload new data with same tables. Call execute_audit("超标准收费", ...) – it must return a DataFrame matching the historical result’s column structure, without calling the AI.

3. Ask the skill list_audit_types() – it must list all rule types, not just the validated one.

4. Check frontend: questions must be rendered as interactive forms, and Markdown must render correctly.

## 9. Final Reminder for the Implementing AI
You are an architect building a business capability compiler. Do not overfit to a single result file. Do not read raw data. Do not call yourself at runtime. Follow the phases in order. Use structured interactions whenever user input is needed. The deliverable is a generic, parameterized Skill, not a script.
