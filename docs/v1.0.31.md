# v1.0.31 项目实现说明与铁律

本文档记录当前项目的整体目标、交互流程、系统边界、交付形态和不可逾越的红线。它不是普通更新日志，而是后续产品、研发和 AI Agent 共同遵守的项目规范。

## 1. 终极目标

本项目的终极目标是：把沉淀在历史业务数据、知识/规则/标准、历史结果中的业务经验，蒸馏成第三方 Agent 能快速安装、快速理解、快速使用、并且可以脱离本平台独立运行的业务场景能力包。

一个合格的业务场景能力包必须满足：

- 第三方接入后能清楚知道：这是什么业务场景、能做什么、需要哪些业务数据、有哪些表和字段、有哪些产出、有哪些工具可用。
- 第三方使用时不依赖蒸馏平台数据库、不依赖平台后端状态、不需要回连平台才能执行业务能力。
- 验证通道与发布后第三方使用的包结构一致，避免“平台内验证可用，第三方安装不可用”的自嗨式结果。
- 能适配任意业务场景，不只服务“医保审计”。医保审计只是验证样例，不是产品边界。
- 用户不需要懂技术也能参与确认。系统提出的问题必须清晰、少量、可选择、有推荐项，不能反复问、不能问明显愚蠢的问题。

核心一句话：

> 平台负责把业务场景蒸馏成标准能力包；能力包一旦发布，就必须能作为独立 Skill/MCP/Docker 能力被第三方快速接入和使用。

## 2. 当前系统定位

系统由四个相互隔离但连续衔接的通道组成：

| 通道 | 作用 | 关键原则 |
| --- | --- | --- |
| 蒸馏工作台 | 上传数据、追踪链路、推导关联、推导流程、生成技能 | 只负责学习和固化业务能力 |
| 验证通道 | 使用生成后的能力包在新数据上验证 | 不能调用蒸馏工具，验证结果反映能力包真实独立能力 |
| 第三方沙盒 | 模拟外部 Agent 平台挂载能力包 | 像 MCP/Skill 宿主一样安装、发现、调用能力 |
| 发布通道 | 构建标准 Skill zip / Docker MCP 镜像并推送 Harbor | 发布物必须可脱离平台运行 |

关键代码入口：

- 后端应用入口：`app/main.py`
- 领域模型与状态机：`app/models.py`
- 蒸馏对话：`app/chat_service.py`、`app/agent.py`、`app/tools.py`
- 数据链路追踪：`app/trace_sampling.py`
- 关联/流程推导：`app/inference.py`、`app/heuristics.py`
- 验证通道：`app/verify_service.py`、`prompts/verification/system.md`
- 第三方沙盒：`app/api/playground.py`、`app/playground_service.py`
- 能力包运行时：`app/scenario_runtime.py`
- MCP Server：`app/mcp_server.py`、`app/api/mcp_http.py`
- 发布构建：`app/release_builder.py`、`app/api/release.py`
- 前端蒸馏页：`frontend/src/views/DistillView.vue`
- 前端表格与追踪展示：`frontend/src/components/TablesPanel.vue`
- 前端发布配置：`frontend/src/components/CapabilityConfigDialog.vue`

## 3. 业务场景状态机

当前业务场景生命周期包含 6 个蒸馏状态和 1 个发布门槛状态：

1. `created`：已创建场景，尚未上传数据。
2. `tables_uploaded`：已上传业务表/知识表/历史结果表，并完成结构解析。
3. `trace_sampled`：已完成数据链路追踪。
4. `relations_deduced`：已推导表关联和字段语义。
5. `flow_deduced`：已推导业务流程节点。
6. `skills_generated`：已生成可复用技能包。
7. `active`：验证通道已确认能力包验证通过，可发布 Docker 镜像。

说明：`active` 是验证通过后的发布门槛，不属于蒸馏本身，但属于完整交付闭环。

## 4. 标准交互流程

### 4.1 新建业务场景

用户在左侧业务场景列表中新建场景，输入名称和描述。场景名称会影响后续发布包命名，例如“医保审计”会被转成类似 `yibao-shenji:1.0.0` 的 Docker 仓库名。

要求：

- 场景名称必须代表真实业务，不要用临时编号作为最终对外名称。
- 场景描述应说明业务目标，但不能写成某个固定表名或某条固定规则的死逻辑。

### 4.2 上传数据

上传阶段只做轻量结构解析：

- 识别有哪些表。
- 记录每张表角色：`input`、`knowledge`、`rule`、`result`、`unknown`。
- 解析表头、字段类型、字段样本值、空值比例。
- 估算行数、列数。
- 保存少量独立样本行用于展示。

上传阶段不允许做的事情：

- 不做数据链路追踪。
- 不做关联推导。
- 不做流程推导。
- 不把某个业务场景的字段、表名、流程写死进平台代码。

原因：上传应该快速、确定、可解释。大表追踪和推导都属于后续显式步骤，不能隐藏在上传里。

### 4.3 数据链路追踪

数据链路追踪是关联推导前的独立步骤。它以历史结果表的一条结果数据为锚点，向业务表和知识表追踪对应样本行，形成“结果行 -> 业务行 -> 知识行”的链路样本。

前端入口：

- “表格 & 字段”页顶部的“数据链路追踪”按钮。
- 右侧对话快捷项“数据链路追踪”。

后端入口：

- `POST /api/scenarios/{scenario_id}/trace-sample`：显式执行并保存追踪。
- `GET /api/scenarios/{scenario_id}/trace-sample`：只读取已保存追踪结果，不现场扫描大表。

追踪结果保存到：

- `Scenario.trace_chain`

追踪结果必须在前端可见：

- 每张表下方可展开“追踪链路样本”。
- 用户能看到该表是否命中、通过什么字段命中、置信度如何、是否退化为随机样本。
- 有问题时用户可以要求 AI 检查/调整。

红线：

- 展开表格详情不能触发隐式追踪。
- 推导关联关系时不能偷偷重跑追踪。
- 流程推导时不能偷偷重跑追踪。
- 追踪慢是可接受的，但必须变成一个用户明确启动、可观察、可修正的步骤。

### 4.4 推导关联关系

推导关联关系必须在数据链路追踪之后执行。

当前推导依据：

- 已保存的 `trace_chain` 链路样本。
- 字段业务语义推断。
- 候选列对的真实值重叠证据。
- LLM 结构化推理，失败时用确定性候选回退。

当前性能优化原则：

- 不再对所有列对盲目全量比较。
- 先用字段名、语义角色、追踪样本、轻量样本值做预筛选。
- 只有值得比较的列对才构建全量去重值集合。
- 输出 summary 中应包含执行明细，例如字段语义耗时、值证据比较数量、预筛跳过数量。

用户确认问题要求：

- 不得重复问同一个问题。
- 不得问“序号和年月/年龄是否有关联”这类明显低质量问题。
- 普通用户能直接做选择，选项应根据问题生成，不只有“确认无误/需要修正”两项。
- 应包含推荐选项，必要时允许“都不是/其他，自行描述”。

人工确认规则：

- 用户明确指出关联修正时，AI 必须调用 `correct_relation` 写回系统。
- 不能只在文本里说“已采纳”。
- 人工确认后的关联必须保留，后续自动推导不能覆盖。
- 修正关联后必须刷新追踪链路。

### 4.5 推导业务流程

业务流程推导使用：

- 表角色。
- 字段语义。
- 已确认/高置信关联。
- 数据链路追踪样本。
- 知识表结构映射。

流程节点必须说清楚：

- `purpose`：该做什么。
- `capability`：能做什么/输出什么。
- `data_in`：输入数据画像。
- `data_out`：输出数据画像。
- `template_kind`：执行模式或算子类型。
- `sql / params`：可执行逻辑或参数。

流程推导成功后，前端必须同步刷新：

- “流程 & 图谱”页能看到流程图谱。
- “产出 & 执行”页能看到产出规格。
- 后续“生成技能”能识别刚推导的流程，不得出现上一步说成功、下一步又说没推导的脱节。

红线：

- 流程推导不得为某个具体业务场景硬编码固定流程。
- 不得假设所有场景都有知识表、规则表、结果表。
- 不得把医保审计流程当成所有业务场景的标准流程。

### 4.6 生成技能

生成技能将流程和领域知识固化成能力包。

内部产物：

- `data/scenarios/<scenario_id>/skills`
- `SKILL.md`
- `mcp.json`
- `manifest.json`
- `main_skill/domain_knowledge.json`
- `main_skill/output_specs.json`
- `main_skill/dispatch_config.json`
- `scripts/query_data.py`
- `utils/scripts/list_knowledge.py`
- `utils/scripts/search_knowledge.py`

技能包必须表达：

- 场景名称。
- 场景用途。
- 什么时候应该使用。
- 什么时候不应该使用。
- 需要哪些业务数据。
- 有哪些表和字段。
- 有哪些输出。
- 有哪些工具。

红线：

- 技能生成后，蒸馏通道结束。
- 蒸馏通道不负责执行验证。
- 执行验证必须切换到验证通道或第三方沙盒。

### 4.7 验证通道

验证通道是能力包独立性的证明。

验证 Agent 只能调用当前业务场景 Skill 包中的工具：

- `describe_capability`
- `list_outputs`
- `describe_schema`
- `execute_skill`
- `list_knowledge`
- `search_knowledge`
- `query_data`

验证通道不能做：

- 不能调用 `deduce_relations`。
- 不能调用 `deduce_flow`。
- 不能调用 `generate_skills`。
- 不能访问平台数据库。
- 不能修改场景元数据。
- 不能自行重新推导关联或流程。

验证数据要求：

- 用户可以上传全新的验证数据到 `verify_uploads/`。
- 验证工具优先使用 `verify_uploads/`，没有才回退到蒸馏阶段 `uploads/`。
- 验证通过表示能力包能在独立数据上工作，而不是复读蒸馏数据。

验证通过后：

- 场景状态可以进入 `active`。
- 发布 Docker 镜像才允许执行。

### 4.8 第三方沙盒

第三方沙盒模拟外部 Agent 平台：

- 左侧能力市场列出当前用户已经生成能力包的场景。
- 用户可以像安装 MCP 一样挂载/卸载能力。
- 通用 Agent 不预置业务知识，而是通过挂载能力自主发现和调用。
- 配置面板展示 Skill、MCP、Docker 等安装方式。

它的价值：

- 提前验证第三方接入体验。
- 检查能力包是否能自描述。
- 检查宿主 Agent 是否知道“这个包是做什么的、怎么用”。

### 4.9 发布通道

发布通道负责把内部 `skills/` 转成标准 release 包。

发布目录：

- `data/scenarios/<scenario_id>/release/<skill_name>`
- `data/scenarios/<scenario_id>/release/artifacts/skill.zip`
- `data/scenarios/<scenario_id>/release/artifacts/toolplane-docker.zip`
- `data/scenarios/<scenario_id>/release/release.json`

支持的安装形态：

| 形态 | 适用第三方 |
| --- | --- |
| Skill 目录/zip | 支持本地 Skill、zip 导入、GitHub Skill 同步的宿主 |
| stdio MCP | 支持 command/args 的本地 MCP 宿主 |
| 远程 MCP URL | 能访问本平台服务地址的第三方，适合托管测试 |
| Docker MCP | ToolPlane 等支持 Docker source 的 MCP 平台 |

Docker 发布流程：

1. 构建镜像：`docker build -t <repository>:<tag> .`
2. 标记镜像：`docker tag <repository>:<tag> harbor.gshbzw.com/skills/<repository>:<tag>`
3. 推送镜像：`docker push harbor.gshbzw.com/skills/<repository>:<tag>`

默认命名：

- 仓库名根据业务场景名称转英文或拼音，例如“医保审计” -> `yibao-shenji`
- 默认 tag：`1.0.0`
- 默认 registry：`harbor.gshbzw.com/skills`

红线：

- 未验证通过不能发布 Docker 镜像。
- 验证通道必须使用 release 包，而不是平台内部 `skills/` 目录。
- release 包不得包含本机绝对路径。
- Docker MCP 包必须能在容器里通过 `/data` 或 `BFE_DATA_DIR` 读取业务数据。

## 5. 发布包的独立运行原则

独立运行的核心代码是 `app/scenario_runtime.py`。它只接收能力包目录作为输入，不依赖平台 store、平台数据库或全局状态。

能力包运行时必须具备：

- `describe_capability`：让第三方知道能力包是什么。
- `describe_schema`：让第三方知道表结构、字段、关联、知识表结构。
- `list_outputs`：让第三方知道能产出什么。
- `list_knowledge/search_knowledge`：让第三方理解可执行的知识条目。
- `execute`：执行结构化产出。
- `query_data`：对任意业务数据做即席 SQL 查询。

重要原则：

- `query_data` 默认不应制造无意义输出文件，除非明确要求保存。
- 第三方 Agent 每个新任务应先调用 `describe_capability` 或 `describe_schema`。
- 能力包只能描述当前挂载的业务场景，不能声称能访问平台上所有场景。

## 6. 项目目录与数据边界

典型场景目录：

```text
data/scenarios/<scenario_id>/
  meta.json                 场景元数据与推导产物
  chat.jsonl                蒸馏通道对话
  verify_chat.jsonl         验证通道对话
  uploads/                  蒸馏阶段原始业务数据
  verify_uploads/           验证通道新业务数据
  skills/                   平台内部能力包
  release/                  第三方发布包
  outputs/                  执行/验证产出
```

数据目录铁律：

- `uploads/` 是蒸馏原始数据。
- `verify_uploads/` 是验证新数据，必须与蒸馏数据物理隔离。
- `skills/` 是平台内部工作产物。
- `release/` 是第三方交付产物。
- `outputs/` 只放明确执行/验证产生的结果，不要让普通查询随手堆垃圾文件。

## 7. 不可逾越的红线铁律

### 7.1 场景无关铁律

平台逻辑必须适应任意业务场景。

禁止：

- 为医保审计硬编码表名、字段名、规则名、流程名。
- 假设每个场景都有知识表。
- 假设每个场景都有规则表。
- 假设每个场景都有历史结果表。
- 假设所有业务都能套后台管理/审计/规则比对一种流程。

允许：

- 把业务场景的表名、字段名、规则文本、流程节点作为“数据”写入能力包。
- 用通用结构表达：表角色、字段语义、关联关系、知识结构、流程节点、产出规格。

### 7.2 一步一停铁律

每条用户消息最多推进一个主步骤：

- `trace_data_links`
- `deduce_relations`
- `deduce_flow`
- `generate_skills`

完成后立刻停下，向用户报告结果，等待下一条明确指令。

禁止：

- 用户说“推导关联”后自动接着推流程。
- 用户说“推导流程”后自动生成技能。
- 有待确认事项时自行进入下一步。

### 7.3 可见性铁律

任何步骤只要返回成功，前端对应区域必须同步看到更新：

- 上传成功：表格和字段刷新。
- 数据链路追踪成功：表格下方追踪样本刷新。
- 关联推导成功：ER 图谱和字段语义刷新。
- 流程推导成功：流程图谱和产出规格刷新。
- 生成技能成功：技能列表刷新。
- 验证通过：发布按钮状态刷新。

禁止：

- 工具说成功但界面没有变化。
- 上一步保存了结果，下一步读取不到。
- AI 文本说已采纳，底层数据没写回。

### 7.4 普通用户确认铁律

系统向用户提问必须符合普通业务用户可理解标准：

- 问题必须少而准。
- 每个问题要有明确上下文。
- 每个问题要给可选答案。
- 选项中应有推荐项。
- 应允许“都不是/其他，自己描述”。
- 重复问题必须去重。
- 明显荒谬的低质量关联问题必须过滤。

禁止：

- 把一堆技术性字段关系直接丢给用户。
- 只给“确认无误/需要修正”两个模糊选项。
- 反复问同一个问题。
- 让用户确认明显不可能的关系。

### 7.5 验证隔离铁律

验证通道必须模拟第三方运行环境。

禁止：

- 验证 Agent 调用平台蒸馏工具。
- 验证 Agent 访问平台数据库。
- 验证 Agent 修改场景状态或元数据。
- 验证时使用一套内部结构，发布时又换一套第三方结构。

必须：

- 验证通道加载 release 包。
- 验证新数据优先使用 `verify_uploads/`。
- 验证通过后才允许发布 Docker 镜像。

### 7.6 发布独立铁律

发布后的能力包必须能脱离平台。

禁止：

- release 包依赖平台 store。
- release 包依赖平台数据库。
- release 包保留本机绝对路径。
- 第三方安装后还必须访问蒸馏平台才能知道能力是什么。

必须：

- 包内包含能力自描述。
- 包内包含运行时脚本。
- 包内包含 schema/manifest/domain_knowledge/output_specs。
- 支持 Skill、MCP、Docker 至少一种主流安装方式。

### 7.7 知识驱动铁律

知识表/规则表可能有成百上千条，每条判断逻辑可能不同。

禁止：

- 蒸馏阶段逐条规则生成固定 SQL。
- 把某条规则字面值写死进节点 SQL。
- 假设知识表就是“规则表”。
- 假设分派键就是“违规类型”。

必须：

- 蒸馏结构，不蒸馏每条规则的死逻辑。
- 固化知识表结构：分派键、条件列、参数列、字段语义映射。
- 运行时由 Agent 读取规则原文，结合 schema 和 `query_data` 现场落地。

### 7.8 大表性能铁律

大表处理必须显式、可解释、可复用。

禁止：

- 上传时隐式扫描大表。
- 展开前端面板时隐式扫描大表。
- 推导关联时重复追踪大表。
- 无筛选地对所有列对构建全量值集合。

必须：

- 大表追踪放在“数据链路追踪”独立步骤。
- 追踪结果保存并复用。
- 关联候选先预筛，再做真实值证据验证。
- summary 中说明实际比较量、跳过量、耗时。

## 8. AI Agent 行为规范

### 8.1 蒸馏 Agent

蒸馏 Agent 只负责：

- 读取元数据。
- 执行数据链路追踪。
- 推导关联和字段语义。
- 推导业务流程。
- 生成技能。
- 根据用户明确修正写回关联。

蒸馏 Agent 不负责：

- 执行新数据验证。
- 查询生产业务结果。
- 发布 Docker 镜像。
- 使用平台外部接口替用户做实际业务操作。

### 8.2 验证 Agent

验证 Agent 只负责：

- 读取能力包描述。
- 读取验证数据。
- 调用能力包工具。
- 执行查询、验证业务结果。
- 汇报验证结论。

验证 Agent 不负责：

- 修改技能包。
- 修改蒸馏结果。
- 重新推导关联或流程。
- 把历史任务和当前任务混在一起执行。

每轮验证回复必须说明：

- 执行了什么。
- 基于什么数据。
- 得到什么结果。
- 是否通过，以及原因。
- 如果失败，失败在哪里。

## 9. 第三方接入标准

能力包对第三方的最低交付标准：

1. 宿主能安装。
2. 宿主能发现能力。
3. 宿主能理解能力。
4. 宿主能知道需要什么数据。
5. 宿主能列出工具。
6. 宿主能执行工具。
7. 宿主能在没有平台的情况下完成业务任务。

不同第三方的接入方式：

| 第三方能力 | 推荐方式 | 当前项目支持 |
| --- | --- | --- |
| 支持 Skill 目录 | Skill 目录/zip | 支持 |
| 支持 GitHub Skill 同步 | 发布 zip 后同步仓库 | 包结构已支持 |
| 支持 stdio MCP | `python run_mcp.py` 或 `python -m bfe_runtime.mcp_server --pkg ...` | 支持 |
| 支持远程 MCP URL | `/api/mcp/...` | 支持托管测试 |
| 支持 Docker MCP | Harbor 镜像 + start command | 支持 |
| 只支持自定义工具文档 | `CAPABILITY.md` / `SCENARIO_CONTEXT.md` | 支持 |

第三方首次接入时推荐调用顺序：

1. `describe_capability`
2. `describe_schema`
3. `list_outputs`
4. `list_knowledge` 或 `search_knowledge`
5. `query_data` 或 `execute`

## 10. 当前 v1.0.31 已具备的关键能力

## 11. 仍需持续守住的风险点

- 不同业务场景还需要继续验证，不能只用医保审计样例证明通用性。
- 大表 Excel 读取仍可能慢，后续可考虑上传后转换为列式缓存或 DuckDB/Parquet 中间层。
- 前端每个步骤的“成功后可见”需要持续做回归检查。
- 结构化确认问题需要持续优化，让业务用户无需理解技术字段也能正确选择。
- 发布包安装体验需要覆盖更多第三方平台，包括仅支持 GitHub、仅支持 zip、仅支持 Docker 的平台。
- 验证通道必须持续确保不会偷用平台内部能力。

## 12. 验收标准

一个业务场景只有满足以下条件，才算真正完成：

1. 用户上传数据后能看到所有表、角色、字段、行列数。
2. 用户显式执行数据链路追踪后，能在每张表下方看到追踪样本。
3. 用户能要求 AI 修正错误链路，修正能写回并刷新。
4. 关联推导后能看到 ER 图谱和字段语义。
5. 流程推导后能看到流程图谱和产出规格。
6. 生成技能后能看到技能列表和能力说明。
7. 验证通道使用 release 包执行，而不是内部目录。
8. 新业务数据能通过验证通道运行。
9. 验证通过后可以发布 Docker 镜像。
10. 第三方接入后能通过 `describe_capability` 明白能力包是什么。
11. 第三方在没有平台数据库的情况下能使用能力包完成业务任务。

## 13. 一句话产品准则

> 任何让能力只能在本平台内跑通、不能被第三方标准安装和独立理解使用的实现，都是无效实现。

> 任何把某个业务场景写死进平台逻辑的实现，都是架构错误。

> 任何成功后前端看不到同步结果、下一步读不到上一步产物的实现，都是流程断裂。

> 任何让普通用户面对技术字段猜关联的问题设计，都是失败的交互。

> 任何验证通道和发布包不一致的实现，都是自嗨验证。
