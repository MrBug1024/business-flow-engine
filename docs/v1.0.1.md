# Project: Business Capability Reverse Engineering & Generalizable Skill Construction

## Core Objective
Given a set of business tables, a **business rule table** (which defines a domain knowledge base, e.g., audit violation types, logic descriptions, examples), and **one or more historical result tables** (which represent *examples* of applying specific rules), reverse engineer the **complete, parameterized business capability** rather than a single rule script.

The final output must be a reusable Skill/Tool that:
- Understands the data schema and table relationships (the "factory blueprint").
- Extracts and generalizes all audit logic from the rule table into a library of executable rule templates.
- Can dynamically execute **any rule** from the library on new incoming data, not just the rule that produced the historical result table.
- Accepts the target rule type as a parameter (e.g., “audit duplicate charges”, “audit over-standard fees”).

## Key Principle: The "Engineer, Not the Script"
The historical result table is merely a **validation sample** for one rule. It must not define the skill's entire capability. Just as a MySQL engineer knows all tables, fields, and relationships, our skill must master the entire domain schema and rule library, and be able to perform arbitrary audits, queries, or updates within that domain.

## Architecture Constraints (Mandatory)
1. **Three-Layer Separation**  
   - **Metadata & Sampling Layer (Python scripts, no AI):** Extract table schemas, field types, cardinality estimates, and only 1–3 sample rows per table. For the rule table, extract structured summaries of each rule (violation type, keywords, logic snippet).  
   - **AI Reasoning Layer (MiniMax-M3):** Works ONLY with the metadata report and rule summaries. It never sees raw data directly. It deduces table relationships, constructs rule templates, and generates SQL/Python logic candidates.  
   - **Verification & Execution Layer (Python scripts):** Runs AI-generated logic on the full original data, compares with historical result tables, and returns difference summaries (matched/missing/extra rows, with a few samples). AI sees only these summaries.

2. **Rule Table as Knowledge Source**  
   The rule table must be explicitly parsed into a list of rule templates (one per violation type or category). Each template includes: required tables/fields, join conditions, filter logic, and aggregation. The AI must ask clarifying questions if a rule’s logic is ambiguous.

3. **Historical Result Table Used Only for Validation**  
   - After deducing a rule template for “duplicate charges”, verify it by executing on historical data and comparing with the provided duplicate-charge result file.  
   - Upon successful validation, the AI must immediately proceed to generalize: produce templates for all other rule types present in the rule table, even if no historical result is provided for them.  
   - If a rule requires external data not present (e.g., a standard price list for “over-standard fees”), the skill must define an interface for that data and alert the user.

4. **Parameterized Skill Interface**  
   The final skill must expose operations like:
   - `list_audit_types()` → returns all available violation types.
   - `execute_audit(violation_type, data_sources)` → dynamically compiles and runs the corresponding rule template.
   It must NOT contain hardcoded logic for only “duplicate charges”.

## Workflow for AI Derivation (Must Follow)
1. Receive metadata reports (generated offline) and the rule table summary.
2. Identify table relationships and build an ER model. Confirm with user via Q&A.
3. Parse the rule table into structured rule templates; request additional field samples if needed (via tools, never full data).
4. Pick the rule corresponding to the provided historical result, generate logic, submit to verification layer, and iterate until match.
5. Once validated, generate logic for all remaining rules in the rule table.
6. Assemble the final Skill with the schema + rule library, providing a generic executor.

## Technology Stack
- Python with langchain>=1.3.0, langgraph==1.2.4, langchain-openai==1.2.1, deepagents==0.6.8
- Frontend: HTML for now
- All heavy data processing via Pandas/DuckDB in scripts, not through the AI model.

## Key Reminder for Implementation
Never feed raw data rows to the AI. Use tools like `extract_metadata`, `get_field_sample`, `execute_and_compare` to bridge the AI and the data. The AI is an architect reading a blueprint, not a worker digging through spreadsheets.
