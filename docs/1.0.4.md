# 业务场景能力包 → 主流标准（MCP 为主 + Skill 兼容）改造方案

## Context（为什么做这件事）

本平台定位是一个"工厂"：把历史数据逆向蒸馏成**任意第三方 Agent 都能零改动挂载**的业务场景能力（Skill 包）。目前蒸馏产物已相当完整（`manifest.json` / `SCENARIO_CONTEXT.md` / `schema.json` / 独立可执行脚本），但存在三个阻碍"交付给第三方"的结构性缺口：

1. **没有对齐任何主流协议**。全库无 MCP、无 Agent-Skills 标准入口。第三方拿到 `skills/` 目录后没有"打开配置面板、粘贴一段配置就能用"的标准路径。
2. **验证通道是"假第三方"**。[verification_agent.py:35](app/verification_agent.py#L35) 的 `build_verify_tools(scenario_id)` 直接伸手进平台内部 `store`，工具硬编码，system_prompt 里还把场景名/技能/执行模式直接喂给 Agent。它证明的不是"第三方零改动能不能用"，而是"平台自己能不能跑"。
3. **缺发现 / 触发判据 / 命名空间隔离**。第三方无法自动知道"这个场景是干嘛的、有哪些工具、什么时候该调用它"；多个场景同时挂载时工具名会撞车。

**目标结果**：蒸馏完成后，每个场景额外产出一套 **MCP 标准能力包**（含能力卡片 + 命名空间化工具 + 粘贴即用的 mcp 配置片段）与 **Agent-Skills 兼容的根 `SKILL.md`**；验证通道 `/verify` 改造成**通用第三方沙盒**——一个"什么业务都不预置知识"的通用 Agent，先在配置面板里"安装/挂载"一个或多个能力包（像加 MCP Server 一样），再靠**自动发现 + 命名空间工具**自行判断何时调用，从而真正验证"零改动挂载、自主发现、自主决策、多场景不冲突"。

已与用户确认的决策：**① MCP 为主 + Skill 兼容；② 验证通道改造为通用第三方沙盒；③ 本地 stdio 为主。**

---

## 架构总览

```
蒸馏产物（每个场景 skills/ 目录）新增：
  mcp.json                能力卡片 + 命名空间化 MCP 工具定义 + 触发判据(when_to_use/not_for)
  SKILL.md（包根）        Anthropic Agent Skills 兼容：frontmatter name/description(触发语)
  mcp_config.example.json 粘贴即用的第三方宿主配置片段（Claude Desktop/Cursor/Cline）

新增平台无关执行核心：
  app/scenario_runtime.py 以「包目录」为输入（不依赖 store）的执行/查询/发现函数

新增可交付第三方的运行时：
  app/mcp_server.py       python -m app.mcp_server  stdio MCP Server（单包 or 多包）

验证通道改造为通用沙盒：
  app/playground_agent.py 通用 Agent（不预置任何业务知识，只有发现工具+已挂载场景的命名空间工具）
  app/api/playground.py   catalog / mounts / config / chat 接口
  web/verify.html         重写为「配置面板 + 通用沙盒」
```

核心解耦原则：`scenario_runtime.py` 只接收**包目录路径**，是 `mcp_server.py`（给第三方）和 `playground_agent.py`（沙盒）的**唯一公共执行源**——这就在代码层面证明了"执行不依赖平台"。

## Part A —— 让"产品"符合标准（`app/skill_builder.py`）

在 `materialize_skills()`（[skill_builder.py:1610](app/skill_builder.py#L1610)）末尾、`_write_manifest` 之后，新增一次 `_write_mcp_descriptor(base, scenario, domain, dispatch, mode, all_skills)`，产出三样：

### A1. `mcp.json`（能力卡片 + 命名空间工具）
- `namespace` = `_slug(scenario.name)`（复用 [skill_builder.py:50](app/skill_builder.py#L50) 的 `_slug`），作为工具名前缀与冲突隔离键。
- `display_name` / `summary`：`summary` 由场景名 + 流程步骤 `purpose` 串联**确定性合成**（无需 LLM，保证一定有值）。
- **`when_to_use`**（触发判据，列表）：从 `required_tables` + `knowledge_table` + 流程目标合成，例如"用户上传了『规则表/结算表/就诊表/项目明细表』并要求做重复收费/违规审计时"。
- **`not_for`**（反触发，列表）：如"与本场景表结构无关的通用数据分析/闲聊"——这是**多场景不冲突**的关键，让宿主 Agent 知道何时**不要**误调。
- `keywords` / `required_tables`（来自 `domain.tables` role=input/knowledge）/ `knowledge_table`。
- `tools`：把 `schema.json` 的 3 个工具重命名为**命名空间化 MCP 工具**：`{ns}__execute` / `{ns}__search_knowledge` / `{ns}__list_knowledge`，再补 `{ns}__describe_schema` / `{ns}__query_data`（对应验证 Agent 已有的 6 个工具，去掉 `data_dir`/`out_dir` 这类第三方不该关心的内部路径参数，改由 runtime 用包内默认目录兜底）。
- `entry`：`{"command":"python","args":["-m","app.mcp_server","--pkg","<pkg_abs>"]}`。

### A2. 包根 `SKILL.md`（Agent Skills 兼容）
frontmatter：
```
---
name: <slug>
description: <summary>。Use this skill when <when_to_use 摘要>。Do NOT use for <not_for 摘要>。
---
```
正文引用 `SCENARIO_CONTEXT.md` + `mcp.json`，说明脚本入口。让"把文件夹丢进 .claude/skills"的 Skill-only 宿主也能靠 description 的触发语做渐进披露。

### A3. `mcp_config.example.json`
粘贴即用片段（stdio）：`{"mcpServers": {"bfe-<slug>": {"command":"python","args":["-m","app.mcp_server","--pkg","<pkg_abs>"]}}}`。同时把它写进 `manifest.json` 的新字段 `mcp`（指针）与 `SCENARIO_CONTEXT.md` 新增一节"六、作为 MCP Server 挂载到第三方"。

> `manifest.json` 现有字段全部保留，只**增量**加 `mcp`/`namespace`/`skill_md` 指针，向后兼容。

## Part B —— 平台无关执行核心（新增 `app/scenario_runtime.py`）

把 [verification_agent.py:35-486](app/verification_agent.py#L35) 内 6 个闭包工具的**执行逻辑**下沉为**以包目录为参数**的纯函数（不 import `store`）：

```
class ScenarioPackage:                    # 读 mcp.json/manifest/domain_knowledge/dispatch
    load(pkg_dir) ；card / namespace / outputs / tables
def list_outputs(pkg, ...) / describe_schema(pkg, ...)
def execute(pkg, output_id, params, data_dir=None, out_dir=None)   # 复用 skill_executor.produce
def search_knowledge / list_knowledge(pkg, ...)                    # 复用 utils/scripts
def query_data(pkg, sql, data_dir=None)                            # 复用 duckdb + load_full_frame_cached
```
- `data_dir` 缺省解析顺序：显式传入 → 包同级 `verify_uploads/` → `uploads/`（沿用 [verification_agent.py:46-56](app/verification_agent.py#L46) 逻辑，但改成相对包目录，不走 `store`）。
- 复用现有 [table_io.py](app/table_io.py) 的 `load_full_frame_cached`（按需加载 + calamine + mtime 缓存，验证通道的性能保障必须保留）。
- `verification_agent.build_verify_tools` **改为薄封装**：内部转调 `scenario_runtime`，保持现有 `/verify` 老接口作为回归基线可用。

## Part C —— 可交付第三方的 MCP Server（新增 `app/mcp_server.py`）

- 依赖：`requirements.txt` 增加 `mcp>=1.2.0`（官方 Python SDK）。**懒加载**：仅 `mcp_server.py` import 它；平台其余部分与沙盒都不依赖 mcp SDK，未安装也能跑（`import` 失败给出清晰安装提示）。
- 传输：`stdio`（`mcp.server.stdio`）。
- 两种运行模式：
  - `--pkg <dir>`：只服务单个场景包（对应 `mcp_config.example.json`）。
  - `--root <data/scenarios>`：扫描所有 `*/skills/mcp.json`，多包同时服务（验证多场景不冲突）。
- 注册的工具：
  - 全局 `list_business_capabilities()` → 返回每个已挂载场景的 `namespace/summary/when_to_use/not_for`（**发现 + 触发判据**入口）。
  - 每个场景注册命名空间工具 `{ns}__execute/__search_knowledge/__list_knowledge/__describe_schema/__query_data`，全部转调 `scenario_runtime`。
- 工具描述里注入该场景的 `when_to_use/not_for`，让宿主模型据此自动判断何时调用。

## Part D —— 验证通道 → 通用第三方沙盒

### D1. 通用 Agent（新增 `app/playground_agent.py`）
- 与现有 `build_verification_agent` 的关键差异：**system_prompt 不含任何具体场景名/技能/执行模式**（现状 [verification_agent.py:508-517](app/verification_agent.py#L508) 把这些直接注入，属于"作弊"）。新 prompt 只说明："你是通用助理；你被挂载了若干业务能力工具；先用 `list_business_capabilities` 了解它们的用途与触发条件，仅在用户诉求匹配某能力的 `when_to_use` 时才调用对应命名空间工具，不匹配就正常回答，不要硬套。"
- 工具集 = `list_business_capabilities`（读所有已挂载包的卡片）+ 每个**已挂载**场景经 `scenario_runtime` 生成的命名空间工具（可多个场景并存）。
- 复用 [agent_guard.py](app/agent_guard.py) 的 `ExcludeBuiltinToolsMiddleware` 与 [verify_service.py](app/verify_service.py) 的流式/轮次生命周期机制（改为面向"已挂载集合"而非单场景）。

### D2. 挂载状态与接口（新增 `app/api/playground.py`）
- 挂载清单持久化：`data/playground_mounts.json`（场景 id 列表）。
- 接口：
  - `GET /api/playground/catalog` —— 列出所有已生成能力包（读各 `mcp.json` 的 `namespace/summary/when_to_use/tools` 摘要），供"能力市场"展示。
  - `GET / POST / DELETE /api/playground/mounts` —— 查看 / 安装 / 卸载挂载场景。
  - `GET /api/playground/mounts/{id}/config` —— 返回该场景 `mcp_config.example.json` 内容 + 能力卡片（配置面板展示"粘贴即用"片段）。
  - `POST /api/playground/chat` —— 通用沙盒流式对话（SSE），基于当前挂载集合构建 `playground_agent`。
- 复用现有 `verify/uploads` 系列接口做每场景测试数据上传（[api/verify.py:54](app/api/verify.py#L54)）。
- `app/api/__init__.py` 注册新 router；老 `/verify/*` 接口保留（回归基线）。

### D3. 前端重写 `web/verify.html`（约 550 行，整页重构）
- 标题改为"第三方 Agent 沙盒 · MCP 能力挂载"。
- 左栏两区：**① 已挂载能力**（含卸载按钮、⚙️ 配置）；**② 能力市场**（catalog，可安装的场景包，显示 summary/when_to_use）。
- **配置面板弹窗**（点⚙️）：展示能力卡片（summary / when_to_use / not_for / 工具清单）+ **粘贴即用的 mcp 配置片段**（含复制按钮）+ 测试数据上传。对齐"像配置一个 MCP 那样"的主流体验。
- 中间：**通用聊天**（不预选场景），调 `/api/playground/chat`；沿用现有工具调用可折叠展示（显示实际命中了哪个 `{ns}__*` 工具、参数、结果）。
- 复用现有 [web/verify.html](web/verify.html) 的 SSE 流式渲染 / `<think>` 折叠 / 消息组件样式，仅改造数据流与左栏信息架构。

## 验证（Verification）

- **单元级**：重新对现有场景执行"生成技能"，确认新增 `mcp.json`（含 `namespace/when_to_use/not_for/tools`）、根 `SKILL.md`（frontmatter 合法）、`mcp_config.example.json` 均生成且 JSON 合法；`manifest.json` 旧字段无缺失。
- **平台无关性**：在**平台代码之外**（另建目录，仅 `pip install -r skills/requirements.txt`）用 `python -m app.mcp_server --pkg <包>` 起 stdio server，跑通 `list_business_capabilities` → `{ns}__describe_schema` → `{ns}__query_data`，证明脱离 `store` 可执行。
- **多场景不冲突**：`--root` 模式挂载 ≥2 个场景，确认工具名因命名空间不撞、`when_to_use/not_for` 区分正确。
- **沙盒端到端**：`/verify` 页面安装某场景 → 查看配置面板的 mcp 片段 → 上传测试数据 → 通用聊天里提出该业务诉求，确认 Agent **先发现后调用**正确命名空间工具并产出；再提一个**该场景 not_for** 的诉求，确认 Agent 不误调。
- **回归**：老 `/verify` 相关接口（`verify/status`、`verify/uploads`）与蒸馏通道 `/` 全流程不受影响。

## 待定/风险

- `mcp` SDK 版本与 API 面（`Server`/`stdio_server`/`list_tools`/`call_tool`）以实装时官方最新为准；用懒加载隔离，未安装不影响平台与沙盒。
- 能力卡片 `when_to_use/not_for` 先走**确定性合成**保证始终有值；后续可选加一步 LLM 润色（不阻塞本次交付）。
- 现有 `/verify` 前端整页重写，样式组件尽量复用，避免回归蒸馏通道。
